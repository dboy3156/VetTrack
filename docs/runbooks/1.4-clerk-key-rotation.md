# Runbook 1.4 — Clerk Production Key Rotation + Git History Scrub

**Severity:** Critical — production Clerk keys (`sk_live_*` / `pk_live_*`) were committed to `.env.production` in git history.

**Affected commits:** `2b66aee1`, `14a64e12`, `907cd1c9`, `82eb42a7`, `86207c57`

**Status:** Must be executed by a repository owner before any external investor or partner is granted read access to the repository.

## Step 1 — Rotate Clerk production keys FIRST

Do this before the history scrub, so the old keys in history are already dead.

1. Log into [dashboard.clerk.com](https://dashboard.clerk.com)
2. Select the VetTrack production instance (`clerk.vettrack.uk`)
3. Navigate to **API Keys** → **Secret keys**
4. Click **Roll secret key** — Clerk will generate a new `sk_live_*` key
5. Copy the new secret key
6. In Railway Dashboard → VetTrack production service → **Variables**:
   - Update `CLERK_SECRET_KEY` to the new value
   - Redeploy the service
7. Verify the production app still authenticates correctly
8. The old `sk_live_5KYn1dNiZjactr3jHEfy5vW8q42G5doqkZfkgjK1Rf` is now invalid — even if someone extracts it from git history, it cannot be used

> **Note:** The public key (`pk_live_*`) is not a secret — it is safe in public git history. Only the `sk_live_*` key is sensitive. Rolling it is sufficient.

---

## Step 2 — Scrub git history

```bash
# From repo root
cd /path/to/VetTrack

# Verify the keys appear in history
git log --all -p --pickaxe-regex -S 'sk_live_5KYn1d' --oneline

# Run filter-repo to remove the string from ALL history
# Replace the actual key value with the literal string to scrub
git filter-repo --replace-text <(echo '==>REMOVED==') \
  --force

# Alternatively, to scrub specific files from history:
git filter-repo --path .env.production --invert-paths --force
```

> Using `--path .env.production --invert-paths` is safer: it removes the file entirely from all history rather than doing text replacement, which can leave partial matches.

---

## Step 3 — Force push to all remotes

```bash
# This rewrites remote history — coordinate with all team members first
git remote -v   # confirm remote name

git push origin --force --all
git push origin --force --tags
```

> All team members must delete their local clone and re-clone fresh. Stale local copies still contain the old history.

## Step 4 — GitHub: request cache purge (optional)

1. Go to the repository settings on GitHub
2. Under **Danger Zone** → consider making the repo temporarily private if it was public
3. Contact GitHub support to request a cache purge if the repo was public and the keys were visible
