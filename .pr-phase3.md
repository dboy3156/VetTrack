## Phase 3 — Persistent scheduler

### Summary
- **`vt_scheduled_notifications`** table (migration `022`) with `type`, `user_id`, `equipment_id`, `scheduled_at`, `sent_at`, `payload` (jsonb), partial index on due rows.
- **Return reminders** scheduled by **insert** (replacing unsent rows for same user+equipment) instead of `setTimeout`; **cancel** deletes unsent rows.
- **`runScheduledNotifications`**: selects due `return_reminder` rows (`scheduled_at <= now()`, `sent_at` null), **claims** each with `UPDATE ... SET sent_at = now() WHERE id = ? AND sent_at IS NULL RETURNING`, then runs existing `processReturnReminderNotification` logic.
- **`startScheduledNotificationProcessor`**: `setInterval` **60s**, started next to hourly smart scheduler after migrations.
- **Idempotency**: concurrent workers lose the claim race when `sent_at` is already set.

### Not included (by design)
- Retry/backoff / payload `attempts` (later phase).

### Stack
Targets **`phase-2-notification-logic`**; merge Phase 2 (and Phase 1) first or rebase onto `main` as needed.
